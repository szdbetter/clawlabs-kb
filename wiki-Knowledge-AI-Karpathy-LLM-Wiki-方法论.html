Karpathy LLM Wiki 方法论

stage/evergreen #domain/ai #type/methodology

原文 Gist：https://gist.github.com/karpathy/442a6bf555914893e9891c11519de94f 作者：Andrej Karpathy 发布日期：2026-04-04 Raw 存档：karpathy-llm-wiki-gist 入库时间：2026-04-07

核心洞察

人类放弃 Wiki 的原因：维护负担的增长速度 > 价值的增长速度。LLM 消除了这个瓶颈。

传统 RAG 模式：每次提问从原始文档检索 → 生成回答 → 答案消失，没有积累。 Karpathy 模式：LLM 持续编译知识到 Wiki → Wiki 不断增长 → 知识复利。

Wiki 是一个持久的、复利增长的产物（persistent, compounding artifact）。交叉引用已经建好，矛盾已经标注，综合分析已经反映了所有已读内容。

三层架构（Three-Layer Architecture）

┌─────────────────────────────────────────────┐
│  Layer 3: Schema（规则层）                    │
│  CLAUDE.md / AGENTS.md                       │
│  定义 Wiki 结构、约定、工作流                    │
│  人类 + LLM 协同演进                           │
├─────────────────────────────────────────────┤
│  Layer 2: Wiki（知识库层）                     │
│  LLM 生成的 Markdown 文件                     │
│  摘要页、实体页、概念页、对比分析、综述            │
│  LLM 写，人类读                               │
├─────────────────────────────────────────────┤
│  Layer 1: Raw Sources（原始资料层）            │
│  论文、文章、图片、数据文件                      │
│  不可变，LLM 只读不写                          │
│  这是事实来源（source of truth）               │
└─────────────────────────────────────────────┘

Layer 1: Raw Sources（原始资料层）

  • 你策展的原始文档集合：文章、论文、图片、数据文件
  • 不可变（immutable）—— LLM 只读不改
  • 这是你的 source of truth

Layer 2: The Wiki（知识库层）

  • LLM 生成的 Markdown 文件目录
  • 包含：摘要、实体页、概念页、对比分析、综述、合成
  • LLM 完全拥有这一层：创建页面、在新来源到达时更新、维护交叉引用、保持一致性
  • 你只读，LLM 负责写

Layer 3: The Schema（规则层）

  • 配置文件（如 CLAUDE.md 或 AGENTS.md）
  • 告诉 LLM：Wiki 的结构、约定、以及在 Ingest/Query/Lint 时的工作流
  • 关键概念：Schema 协同演进（co-evolution）—— 你和 LLM 在使用过程中共同迭代优化这个规则文件
  • 这是把 LLM 从通用聊天机器人变成专业 Wiki 维护者的关键

三个核心操作（Three Operations）

1. Ingest（录入）

将新来源放入 Raw 集合 → 告诉 LLM 处理它： - 读取来源，与你讨论关键要点 - 写摘要页到 Wiki - 更新 index.md（索引） - 更新所有相关的实体页和概念页 - 追加条目到 log.md - 一个来源可能触发 10-15 个 Wiki 页面的更新 - 可以逐个录入（深度参与）或批量录入（轻度监督）

2. Query（查询）

对 Wiki 提问 → LLM 搜索相关页面 → 综合回答： - 输出形式多样：Markdown 页面、对比表格、幻灯片（Marp）、图表（matplotlib）、canvas - 关键洞察：好的回答可以回存 Wiki 变成新页面 - 查询本身也在积累知识，你的探索和来源一样产生复利

3. Lint（健康检查）

定期让 LLM 检查 Wiki 健康状态： - 发现页面间的矛盾 - 标记已被新来源取代的过时声明 - 找孤儿页面（没有入站链接） - 找被提及但缺少专属页面的重要概念 - 找缺失的交叉引用 - 建议可用网络搜索填补的数据空白 - 建议下一步该研究什么和寻找什么来源

index.md 和 log.md 的角色

index.md —— 内容导向

  • Wiki 的目录/索引：每个页面的链接 + 一行摘要 + 可选元数据
  • 按类别组织（实体、概念、来源等）
  • LLM 每次 Ingest 都更新它
  • 回答查询时，LLM 先读 index.md 定位页面，再深入具体内容
  • 在中等规模（约100来源，数百页面）下效果很好，无需嵌入式 RAG 基础设施

log.md —— 时间导向

  • 追加式记录：什么操作在什么时间发生
  • 条目格式：## [2026-04-02] ingest | Article Title
  • 可用 unix 工具解析：grep "^## \[" log.md | tail -5
  • 提供 Wiki 演化的时间线
  • 帮助 LLM 了解最近做了什么

Schema 协同演进（Co-evolution）

Schema 不是一次性设定的静态文件，而是： - 人类和 LLM 在使用过程中共同迭代 - 随着你了解到什么在你的领域有效，不断优化 - 记录约定、页面格式、工作流规则 - 这是让系统越来越智能的关键机制

适用场景

  • 个人成长：目标、健康、心理、自我提升的结构化追踪
  • 研究：深入一个主题数周/数月，逐步构建有演化论点的综合 Wiki
  • 读书：按章节录入，建立角色页、主题页、情节线索页（类似 Tolkien Gateway 粉丝 Wiki）
  • 团队/商业：LLM 维护的内部 Wiki，输入来自 Slack、会议记录、项目文档、客户通话
  • 竞争分析、尽职调查、旅行规划、课程笔记、爱好深挖

哲学渊源

这个想法在精神上与 Vannevar Bush 的 Memex（1945）相关 —— 一个个人的、策展的知识存储，文档之间有关联轨迹。Bush 的愿景比 Web 的实际走向更接近这个模式：私密的、主动策展的、文档之间的连接与文档本身同样有价值。他无法解决的部分是谁来做维护。LLM 解决了这个问题。

相关实现

Graphify — 从 Wiki 到 Knowledge Graph

Graphify（⭐ 3.2k Stars）是第一个将本方法论工程化实现的开源项目，发布一周即获得广泛关注。核心进化：

  • 从 Wiki 到 Graph：不是平面 Markdown，而是 NetworkX 知识图谱 + Leiden 社区检测
  • 双轨提取引擎：代码走 tree-sitter AST（零 Token），文档/图片走 LLM 语义提取
  • 信任审计链：三级置信度标签（EXTRACTED / INFERRED / AMBIGUOUS）
  • 71.5x Token 压缩：一次编译，无限次高效查询

详见 Graphify-LLM知识图谱工具。
