Karpathy LLM Wiki 知识管理方法论

stage/evergreen #domain/ai #stage/evergreen

来源：微信公众号《Karpathy 开源了 Agent + Obsidian 个人知识库，超级有启发》 原文：https://mp.weixin.qq.com/s/FG2gEdgZwpMkA3ApQMksnw Gist：https://gist.github.com/karpathy/442a6bf555914893e9891c11519de94f 入库时间：2026-04-07

核心思想

不要把 LLM 当搜索引擎，而是让它像程序员维护代码库一样，持续维护一个 Markdown 知识 Wiki。

关键区别： - RAG（旧方式）：每次提问 → 从原始文档检索 → 生成回答 → 答案消失，没有积累 - Karpathy 方式：LLM 持续编译知识 → Wiki 不断增长 → 知识复利

三层架构

Layer 1: Raw Sources（原始资料层）
  → 论文、文章、图片、数据文件
  → 只读，LLM 不写这层

Layer 2: The Wiki（知识库层）
  → LLM 生成的 Markdown 文件
  → 摘要页、实体页、概念页、对比分析、综述
  → LLM 写，人类读

Layer 3: The Schema（规则文件）
  → CLAUDE.md / AGENTS.md
  → 告诉 LLM 如何组织 Wiki、遵循什么约定
  → 人类和 LLM 随使用迭代优化

三个核心操作

Ingest（录入）

丢一个新文件 → LLM 处理： - 提取关键信息 - 写摘要页 - 更新相关实体/概念页（1个来源 → 触发10-15个页面更新） - 更新索引

Query（提问）

对 Wiki 提问 → LLM 搜索相关页面 → 综合回答 - 输出可以是 Markdown、表格、Marp 幻灯片、matplotlib 图表 - 好的回答可以回存 Wiki，变成新页面（这是关键：查询本身也在积累知识）

Lint（体检）

定期让 LLM 检查 Wiki 健康状态： - 找页面间的矛盾 - 标记过时信息 - 找孤儿页面（没有入链） - 找缺失的交叉引用 - 建议下一步该研究什么

实际工作流

Obsidian（IDE）←→ Claude Code（程序员）←→ Wiki（代码库）

用户同时开着 Agent 和 Obsidian：
  LLM 编辑 Wiki → 用户在 Obsidian 实时浏览结果
  跟着链接点看、看图谱视图、读更新后的页面

工具推荐： - Obsidian Web Clipper：网页 → Markdown（快速录入） - index.md：内容目录 - log.md：时间线日志 - qmd：Wiki 变大后的搜索引擎（已在 ClawLabs VPS 安装 ✅） - Dataview 插件：基于元数据动态查询 - Marp：Wiki 内容直接生成幻灯片

规模参考

约 100 个来源、几百个页面时，不需要向量数据库或复杂 RAG： → LLM 先读 index.md 定位页面 → 再深入具体内容 → 效果很好

为什么 LLM 比人类更适合维护 Wiki

人类放弃 Wiki 的原因：维护负担增长速度 > 价值增长速度

LLM 的优势： - 不会厌倦重复的簿记工作 - 不会忘记更新交叉引用 - 可以一次性修改 15 个文件 - 维护成本接近零
