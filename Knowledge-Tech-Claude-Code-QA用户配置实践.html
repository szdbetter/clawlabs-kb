Claude Code qa用户配置实践

背景

老板希望在qa用户下运行Claude Code，实现多用户隔离（root用claude，qa用claude）。

2026-03-29 配置过程记录

[来源: Claude-Code 会话记录]

问题1：qa用户找不到claude命令

  • 现象：su - qa 后 claude command not found
  • 解决：软链 ln -s $(which claude) /usr/local/bin/claude 或配置PATH

问题2：claude在qa下要求重新登录

  • 现象：qa用户下运行claude，提示是新用户需要OAuth授权
  • 原因：Claude Code的认证信息存储在用户目录下（~/.claude/），qa用户无法读取root的认证
  • 解决方案B（选定）：将root的~/.claude/projects所有目录权限开放给qa用户，不另外安装

问题3：qa用户想软链projects目录

  • 方案：su - qa 后，~/.claude/projects 软链到 /root/.claude/projects
  • 约束：qa用户需要有创建目录/软链的权限

问题4：qa进入claude提示已登录（Welcome back Jimmy！）

  • 关键发现：qa用户已能进入Claude Code（认证共享成功）
  • 遗留问题：/login 仍有触发，需检查认证文件权限

defaultMode选项说明

  • default：标准模式
  • acceptEdits：自动接受文件编辑
  • autoEdit：自动编辑（不确认）
  • bypassPermissions：跳过所有权限检查（危险，仅受信任环境）

CLAUDE.md同步

  • 已将root的CLAUDE.md复制到qa用户环境

2026-04-01 补充：目录权限问题根因排查

[来源: Claude-Code binance/content 项目会话 | 2026-04-01]

Linux组权限优先级陷阱

  • 现象：qa用户在root组，但仍无法访问 /root/.openclaw/workspace/projects/
  • 根因：/root/.openclaw/ 权限 0705，group(root) = ---，Linux优先匹配group权限 ---，不回退到other！
  • 修复：find /root/.openclaw -type d -exec chmod g+rx {} \;
  • 教训：Linux ACL中，如果用户在某个group中，会以group权限为准，even if other有更宽松权限

软链接方案完成（全局共享认证）

  • /home/qa/.claude/projects → /root/.claude/projects（全局软链，自动同步新目录）
  • /home/qa/.claude/skills → /root/.claude/skills
  • qa用户现在可以正常 /rename、/resume、/skills

实际工作效果

  • qa用户能访问root下的全部Claude Code项目
  • 老板在qa用户下开发Binance相关脚本（数据分析/策略）
  • 隔离性：qa运行环境独立，root核心配置不受干扰
