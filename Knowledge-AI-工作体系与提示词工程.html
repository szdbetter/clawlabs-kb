工作体系与提示词工程

来源：Discord Thread #工作体系/提示词完善 (1482904446588555367, 498条) 提取日期：2026-03-20

一、AI工作体系设计哲学

核心文件架构

OpenClaw的工作体系由4个核心文件驱动： - SOUL.md — CEO状态机，定义角色、决策逻辑、STATE 0→3 - AGENTS.md — 执行法则，分支A(CEO模式)/分支B(执行节点) - TOOLS.md — 工具规范，检索优先级、子智能体防污染协议 - USER.md — 用户设定，不作为方法论修改

双环境架构

  • OpenClaw环境（Discord对话）：skill作为"专家知识"指导AI行为模式
  • Claude Code环境（终端编程）：skill作为plugin提供Hook和工具
  • 两个环境共享同一份skill代码，网关重启后自动生效

二、三套外部体系对比分析

gstack（Garry Tan）

  • 核心哲学："Boil the Lake"——AI让完整性边际成本→0
  • 优势：13个角色矩阵(CEO/EM/Designer/QA/RE)，双刻度ROI估算，100ms/命令的真实Chromium浏览器
  • 劣势："Boil the Lake"与ROI优先哲学冲突；无长期记忆系统；仅适合编程场景
  • OpenClaw适配度：🥈 次之（编程专用）

Planning-with-Files（OthmanAdi）

  • 核心哲学：Context=RAM(易失)，Filesystem=Disk(持久)
  • 优势：三文件模式(task_plan/findings/progress)通用任何任务类型；Hook持久化机制；96.7%的pass rate提升
  • 劣势：无角色分工，与SOUL.md状态机概念重叠
  • OpenClaw适配度：🥇 最高

Superpowers（14个skill）

  • 核心哲学：TDD优先、证据>声称
  • 优势：TDD闭环内置，git-worktree隔离，14个独立skill模块化
  • 劣势：过度工程化，7步流程仅适合纯编程场景
  • OpenClaw适配度：🥉 最低（仅编程子任务）

吸收策略

  • Planning-with-Files：只引入Hook持久化机制，叠加在现有task.md之上
  • gstack：借用双刻度时间估算（人类vs AI时间），工具层复用
  • Superpowers：仅在编程场景触发（brainstorming→writing-plans→TDD闭环）

三、Skill系统设计与管理

Skill加载机制

  • OpenClaw扫描workspace/skills/**/SKILL.md路径
  • 不跟随symlink（需要真实拷贝）
  • system prompt字符上限：~30,000字符
  • 57个eligible skill中只有约23-42个进入system prompt

Skill自动匹配原理

用户正常发需求，AI根据skill的description字段自动匹配：

"分析一下特斯拉" → investment-masters
"帮我抓@VitalikButerin的推文" → twitter-crawler
"做一个价格监控" → sp-brainstorming → sp-writing-plans → planning-with-files 链式触发

Skill归档策略

归档到_archive/目录的原则： 1. 低频使用（polymarket-fast-loop, polymarket-weather-trader） 2. 不适配当前环境（stt, wechat） 3. 仅Claude Code有效（sp-using-superpowers, sp-subagent-driven-development） 4. 子skill本应按需加载而非全量展开（gstack的14个子skill）

恢复方式：mv _archive/xxx skills/ + 网关重启

在TOOLS.md维护归档索引，确保知道有什么可用。

gstack子skill问题

  • gstack 14个子skill占~700 token
  • 触发条件极其特定（"review a PR"、"ship to production"），日常不命中
  • 解决方案：归档gstack子skill，只保留主skill

四、Context优化实践

优化前后对比（2026-03-20）

  • AGENTS.md：4,379B → 1,009B（-77%）
  • TOOLS.md：2,902B → 638B（-78%）
  • 活跃skill：26 → 18个
  • 总压缩率：-30%

删除的内容类型

  1. AGENTS.md中删除：与SOUL.md重复的verification铁律、TDD流程、brainstorming细节
  2. TOOLS.md中删除：Skill使用场景指南表（与skill description完全重复）
  3. 归档的8个skill：polymarket-fast-loop/weather-trader、stt、wechat、twitter-browser-harvest、obsidian-sync、sp-using-superpowers、sp-subagent-driven-development

优化原则

  • 不重复：同一规则不出现在2-3个地方
  • 技术细节由skill自身承载，核心文件只做高层决策
  • SOUL.md暂不动：6KB是大头，状态机是核心逻辑

五、Superpowers Skill使用场景

Skill 触发场景 举例
sp-brainstorming 任何创造性工作前 需求澄清+设计
sp-writing-plans 多步骤任务拆解 生成原子化任务列表
sp-test-driven-development 写代码前 RED→GREEN→REFACTOR
sp-systematic-debugging 遇到bug时 复现→假设→验证→修复→回归
sp-verification-before-completion 宣称完成前 强制跑验证命令，拿到证据才能说完成
planning-with-files 复杂项目(>5步) 自动创建三文件追踪进度

编程任务标准流程

新功能：gstack /plan-ceo-review + superpowers TDD
Bug修复：sp-systematic-debugging + sp-verification-before-completion
跨天研究：planning-with-files 三文件
代码审查：gstack /review
QA测试：gstack /qa (真实Chromium)
发布：gstack /ship

六、验证铁律

NO COMPLETION CLAIMS WITHOUT FRESH VERIFICATION EVIDENCE

验证5步： 1. IDENTIFY — 确认验证命令 2. RUN — 执行命令 3. READ — 检查输出 4. VERIFY — 确认结果 5. DONE — 标记完成附证据

红旗词汇（见到立即停下跑验证）： - "应该没问题" - "大概可以" - "看起来正确"

七、关键教训

  1. symlink问题：OpenClaw不跟随symlink，必须用真实目录拷贝
  2. gstack体积问题：browse/dist/占199MB编译产物，需清理
  3. context污染：规则重复出现在多个文件=token浪费
  4. skill加载限制：字符上限导致超出部分被截断，归档低频skill是正解
