Cron 任务调优与日报自动化

来源：[OpenClaw会话] Discord #市场日报 频道 + #obsidian 频道 2026-04-05~09 入库时间：2026-04-09

核心内容

Cron 超时根因分析

现象：市场日报 cron 任务（JobId: 69703a0b）连续超时 10 次（360 秒限制）。

根因：子 agent 串行执行数据抓取 + 分析 + 发布，每一步都要等前一步完成。单步 API 调用（Binance + Alternative.me + Dexscreener + xreach Twitter）本身不慢，但串行累加超出超时。

解决方案： 1. 脚本化抓取：将数据抓取逻辑写成独立脚本放入 workspace，cron 调用脚本而非让 agent 逐步执行 2. 并行化：同时发起多个 API 请求，合并结果后再分析 3. 增大超时：对复杂任务将 timeout 从 360s 提高到 900s

效果：优化后 10 秒完成（从 360 秒超时降到 10 秒），消息正常发出。

obsidian-daily-digest cron 超时

不同于市场日报：此问题的根因是 LCM 上下文压缩阶段的 summarization 超时，不是数据抓取超时。

解决：调高 LCM 压缩阈值 lcm.threshold 0.85（详见 LCM-长上下文压缩运维指南）

日报内容迭代决策

V1 → V2 的核心变更：

用户反馈第一部分（OpenClaw/AI 动态）用 EXA 搜出的内容对加密投资决策零价值，从不点开看。

维度 V1 V2
结构 OpenClaw/AI动态 + 项目追踪 市场脉搏 + 项目追踪
市场脉搏 无 BTC/ETH 价格 + 恐贪指数 + 清算 + 资金费率 + 一句话判断
项目追踪 罗列数据 基本面判断（⬆️/↗️/➡️/↘️/⬇️ + 一句话结论）
数据源 EXA API Binance API + Alternative.me + Dexscreener + xreach Twitter
输出格式 详细报告 一句话结论 + 关键数据

判断标准框架： - 官方重大公告 = 基本面变化 - KOL 背书 = 情绪催化（非基本面） - 链上异常（+30%）= 需注明原因 - 无官方更新 + 价格上涨 = 纯市场情绪

Cron 使用原则

cron 是宝贵资源，只用于固定时间事件（日报、告警心跳）。周期性查询（额度刷新、健康检查等）纳入 Dashboard 脚本统一管理。每新增 cron 前必须先问：能不能放进 dashboard 脚本？

关键教训

  1. 先手动跑通再配 cron：复杂任务先 manual run 验证全流程，再写入 cron
  2. status=error 不等于失败：cron 报错但 delivered=true 说明消息已发出，是假告警
  3. xreach 认证：Twitter 数据抓取需要配置 auth token，未配置会静默失败
