Elon-Poly 交易板系统进展

系统架构

  • 监控系统：/api/elon/research-monitor + /elon2 页面
  • 双盘口视图：同时展示覆盖当前日期的两个 Elon 推文盘口
  • 实时告警：当 manual_trade_plan.ready = true 时，页面展示红色横幅 + 浏览器通知

🆕 2026-03-29 更新

[来源: workspace/projects/elon-poly / Claude Code]

Live Monitor 切换与加固

  • 系统不再依赖 events?active=true 单一路径
  • 改为按真实滚动规律探测周二/周五 7天窗口 slug
  • 已发现活跃事件：316317（March 31 - April 7）、301949（March 27 - April 3）、291733（March 24 - March 31）

双盘口交易员视图重构

  • /elon2 从技术监控卡片重构为交易员双盘口表格
  • 新增 build_adjacent_pair_board_snapshot() Python数据层
  • 表格列：数量区间、Yes价格、No价格、相邻偏差、流动性代理、最近成交、快照时间
  • 优先展示覆盖当前日期的盘口（301949 + 291733），未来盘口316317不作默认第一视角

浏览器告警机制

  • ready = true 时弹出系统通知（需浏览器权限）
  • 信号去重：基于 event_id + pair + spread + snapshot_time 生成指纹存入 localStorage

设计规范文档

  • 参见：projects/elon-poly/docs/superpowers/specs/2026-03-29-elon2-trader-board-design.md
  • 实现计划：projects/elon-poly/docs/superpowers/plans/2026-03-29-elon2-trader-board-implementation-plan.md

🆕 2026-03-31 更新

[来源: Codex sessions / elon-research / daily-digest]

elon-research 研究框架结论

  • 8周真实 Polymarket 数据已接入并标准化（事件：208544~278377）
  • 代码骨架完成：ingest → normalize → backtest 全链路打通
  • 27个测试全部通过（pytest -q）

数据限制确认

  • data-api /trades 在 offset > 3000 返回400，8/8抽样 market 均触发
  • 市场微结构基线存在单 market 截断风险
  • market 级分页修复路径（slug/conditionId 过滤）已确认无效，已回滚

当前阶段状态（阶段检查点 2026-03-28）

  • 阶段性结论：当前数据条件下无可执行 edge，结论需保守解释
  • 不要宣称已发现可交易 edge
  • 下一步：获取逐条历史推文 A 级来源，或找到成交完整性替代路径

2026-04-09 更新（Codex 会话扫描）

[来源: Codex sessions 019d322e / 019d3233 / 019d3339]

Goldsky Subgraph 替代数据源

作为 data-api /trades 截断问题的替代方案，引入 Goldsky orderbook subgraph：

  • 新增 CLI 命令：ingest-subgraph-market、ingest-subgraph-event
  • 数据路径：datasets/raw/polymarket_subgraph/event_<ID>_market_*.json → datasets/normalized/event_<ID>_subgraph_trades.json
  • 首轮结果：事件 278377 的 subgraph 数据约 6084 行，覆盖 brackets 260-319
  • 批量抓取：已对 8 个事件（208544~278377）执行 ingest-subgraph-event --top-brackets 3 + normalize

多代理并行开发模式

elon-research 开发采用了 Codex多代理并行审查模式： - 实现代理（gpt-5.4-mini）：限定 3-5 个文件范围，TDD 驱动 - Spec 审查代理 + Quality 审查代理并行审查 - Task 5→6→7 按此模式逐步推进 - 每个 Task 有严格的 DONE/BLOCKED 输出格式

xTracker 推文数据接入

  • 批量接入 8 周 xtracker tracking ingest+normalize
  • 事件映射：208544→6e57926f, 212454→9cd52ebd, ... 278377→d861bacb
  • 生成 datasets/normalized/tracking_<id>_tweets.json
  • 目标：补充 A 级逐条历史推文时间线

研究工程化体系

完整的研究工具链已落地： - CLI：ingest / normalize / backtest / ingest-subgraph-event / ingest-subgraph-market - 评估：adjacent_pair.py 相邻配对分析 + reporting.py 中文摘要报告 - 监控：shadow_monitor_service.py 每轮 iteration 实时刷新 summary/json/md - 文档：experiments/README.md + datasets/reports/README.md 记录实验约束

关键教训（从 Codex 会话提取）

  1. 不要把弱验证结果包装成已发现可交易 edge — 保守解释是铁律
  2. proxy_tweet_windows 已接入但无可辩护 edge — 修正数据偏差后，跨周统一参数和 proxy window 都未观察到信号
  3. session bootstrap 文档是跨会话延续的关键 — 每次会话结束前写 checkpoint + handoff 文档
