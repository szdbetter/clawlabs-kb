产品方法论

← 返回首页

[!abstract] 核心洞察 "收口四条件"哲学的本质是：任何可以被任意宣称"完成"的流程，就是一个会持续腐烂的流程。把完成标准从协议升级为硬门槛，是产品化思维最重要的一步。

产品方法论 #AI应用 #数字分身 #MVP #自动化 #垂直场景

一、"收口四条件"哲学

来源：KOL数字分身项目 的关键转折

问题：卡兹克的三件套文件生成了，但没写入REGISTRY和长期记忆，导致跨Channel查询时"找不到"。

根因：旧SOP把"三件套生成"当成完成，缺少强制最后一步登记。

强制收口四条件（数字分身SOP）

只有同时满足以下4条，才允许宣布"已完成"：
1. 三件套生成（raw.md + advisor_profile.md + avatar_prompt.md）
2. 写入 REGISTRY.md（登记表）
3. 写入/补强长期记忆（OpenViking）
4. 更新 STATUS.md（状态板）

缺任意一条 = 未完成，禁止回复"已完成"。

这个哲学的普适性

这不只是数字分身项目的规则，是所有AI自动化项目的反腐败机制：

开发SOP：CODE → TEST → DEPLOY ← 常见偷懒点
         ↓          ↓        ↓
正确SOP：CODE → TEST → VERIFY_EVIDENCE → DEPLOY → CONFIRM
                              ↑
                    "应该没问题"是红旗词，必须有证据

知识库写入SOP：WRITE → LINK → MEMORY → NOTIFY
               ↓        ↓        ↓         ↓
            不能只写  必须双链   必须入库   必须告知

核心洞察：把"完成条件"从可协商的协议升级为不可绕过的门槛，就是产品化思维。

二、AI + 垂直场景的切入模式

闲鱼票务：从工具到系统

MVP轨迹：
人工接单 → AI辅助报价（Gemini Vision识别截图）
         → 自动报价（底价×1.08）
         → 付款检测+Discord推送
         → 自动改价（CDP操作编辑页）
         → WebSocket实时推送（完全脱离物理节点）

关键洞察：
- 闲鱼没有个人开发者API → 必须逆向IM协议
- 真实业务约束倒逼技术创新

→ 详见 闲鱼电影票自动化项目

电商客服：从工具到工作流

来自 电商行业AI应用调研 的核心结论：

不是另一个智能客服SaaS，而是商家随身携带、在消息里直接办事的运营Agent。

四个筛选条件（高ROI场景）：

1. 本来就在消息里发生（天然入口）
2. 不是只聊天，而是"查数据+做动作+发通知"
3. 高频、重复、人工耗时重
4. 适合AI先做、人再确认（不能完全无人化）

正确的商业化顺序：

先卖方案 → 再卖模板（跨境/报价/售后模板）
→ 再卖硬件（已配置终端+行业版套餐）
→ 最后做私有化/行业版

KOL数字分身：从内容到顾问系统

错误定位：做"像某某说话"的玩具
正确定位：建立可调用的顾问资产

价值主张：
"当我遇到一个问题时，我希望能立刻切到某个人的思维模型里，
 听听'他会怎么判断'。"

规模化路径：
单个顾问 → SOP标准化 → 批量制造顾问的流水线

→ 详见 KOL数字分身项目

三、从MVP到自动化的演进路径

通用演进模型

Phase 0：手动验证（最重要！）
├── 目标：验证核心假设，而非构建系统
├── 方法：人工操作，记录痛点和成功率
└── 决策门：胜率/ROI是否值得自动化？

Phase 1：工具化（减轻人力）
├── 自动化重复步骤
├── 保留人工决策关键节点
└── 产出：可用但需人工干预的工具

Phase 2：系统化（自动化闭环）
├── 完整自动化流程
├── 异常处理和告警
└── 产出：能够独立运行的系统

Phase 3：产品化（可复制交付）
├── SOP文档化
├── 标准化接口和配置
└── 产出：他人可以使用的产品

SmartMoney监控的正确演进

❌ 错误路径：直接构建完整自动化系统
✓ 正确路径：
  Phase 0：手动用GMGN+BullX跟单2周，记录胜率
  Phase 1：基于open-sol-bot搭MVP（胜率>45%才进入）
  Phase 2：Helius gRPC实时 + AI SM评分（Phase 1 ROI为正才进入）

教训：严禁不做竞品调研就启动项目；严禁先写代码再做需求分析。

四、课件制作即产品化思维

OpenClaw课件与教程 的产品化洞察

课件结构即产品结构：

P1 封面 → 品牌定位
P2-P6 教育层 → 认知重塑（高级打字机 → 数字员工）
P9 目标用户 → 三类用户分层
P10 时机判断 → "2026 Agent元年"制造紧迫感
P11 商业案例 → 真实落地降低购买风险
P12 收口页 → 行动召唤

核心金句（产品Slogan）：
"下一个时代你要么拥有数字员工，要么成为别人的工具"

并行赛马制作流程：

CEO骨架版（定框架）
    ↓
并行国产模型出稿（Qwen/GLM双路赛马）
    ↓
验收选优
    ↓
锁定文案版（ppt_copy_final.md）
    ↓
生成PPTX

关键认知：PPT课件的本质是产品说明书——每一页都是一个产品决策，每个决策都需要有意图。

五、知识库即产品

Obsidian系统的产品化设计

捕获层（低摩擦）：AI对话自动提炼，无需手动整理
存储层（结构化）：标准frontmatter + 分类目录
关联层（高价值）：<span class="wikilink-missing" title="未找到笔记">双向链接</span> 形成知识图谱
检索层（可用性）：qmd语义搜索 + rg全文搜索
展示层（可视化）：Mac Obsidian Graph View

核心原则：

"Obsidian是容器，OpenClaw是往里面填内容和整理内容的引擎。" "知识库是生长出来的，不是整理出来的。" "捕获的门槛要极低。说几句话，AI帮整理——这才是能持续的模式。"
