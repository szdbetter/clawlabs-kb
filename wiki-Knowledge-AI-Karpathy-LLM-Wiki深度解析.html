Karpathy LLM Wiki 深度解析

stage/evergreen #domain/ai #stage/evergreen

来源：微信公众号《快速上手：Karpathy 的 LLM Wiki，到底比传统知识库多了哪一层》 原文：https://mp.weixin.qq.com/s/QaqYjLejQm9-2eNY_wU4eQ 入库时间：2026-04-07 相关笔记：Karpathy-LLM-Wiki知识管理方法论

核心差异：比 RAG 多了什么

维度 传统 RAG LLM Wiki
主要动作 查询时检索原文片段 先把知识编译成结构化 wiki
中间产物 很薄或没有 持续存在的 Markdown 知识层
问答后沉淀 停在聊天记录 好的结果回写成新页面
模型角色 检索后回答 持续维护知识系统
知识积累 每次重新发现 复利增长

一句话：LLM Wiki 比传统 RAG 多出来的是"原始资料和最终回答之间，一层会被持续维护的知识中间层"。

index.md vs log.md：两个关键文件

Karpathy 把索引拆成了两个特殊文件，分工不同：

文件 功能 类比
index.md 内容地图，按类别列出页面 + 一行摘要 软件项目的"目录索引"
log.md 时间线流水，记录每次 ingest/query/lint 软件项目的"变更记录"
  • index.md 管空间（什么内容在哪里）
  • log.md 管时间（什么时候做了什么）

LLM 查询时先读 index.md 定位相关页面，再往下钻——这就是为什么中等规模（100篇/40万词）不需要复杂 RAG 的原因：索引已经大幅缩短检索路径。

File Back（回写）：知识复利的关键

最容易被忽略的一步：好的查询结果可以回写成新页面。

提问 → LLM 搜索 Wiki → 生成回答
                              ↓
                      回写成新 Wiki 页面
                              ↓
                      下次查询的新上下文

一旦这个闭环建立： - 问答不再只是消耗上下文，也在生产上下文 - 知识不会蒸发在聊天记录里 - Wiki 随使用自然增长

Farzapedia：最具体的案例

Farza 让 LLM 读取了 2500 条个人材料（日记、Apple Notes、iMessage），生成了 400 篇相互链接的个人百科文章。

关键洞见：这个 Wiki 不是为他建的，是为他的 Agent 建的。

实际用法： 1. Claude Code 从 index.md 开始 2. 一层层钻到需要的页面 3. 综合出答案（比如：找最近喜欢的图片+电影+审美线索 → 生成 landing page 文案）

→ Wiki 成为 Agent 的长期工作底稿，而非人类的记忆容器

Karpathy 转发时补的4个点： - file over app（文件优于应用） - BYOAI（自带 AI） - 数据主权 - 知识私人化

规模与 RAG 的关系

Karpathy 说"100篇/40万词不需要复杂 RAG"，前提是：先把原始资料整理成结构化知识层（index + summaries + concept pages）。

不是说 RAG 不重要，而是： - 中等规模 + 主题聚焦 → index 导航够用 - 大规模 + 主题分散 → 还是需要向量检索 + 校验

治理：最难的部分

人类放弃 Wiki 的原因：维护负担增长速度 > 价值增长速度

LLM 解决了维护问题，但带来了新的治理挑战： - 页面间的矛盾谁来发现？（→ Lint 操作） - 过时信息谁来标记？ - 哪些结论需要人工验证？

Karpathy 的原则：LLM 维护，人类验证（"human owns verification"）

完整工作流

raw/ 目录（只读，原始资料）
    ↓ Ingest（LLM 处理）
wiki/ 目录（LLM 写，人读）
    ├── index.md（内容导航）
    ├── log.md（操作时间线）
    ├── summaries/（摘要页）
    ├── concepts/（概念页）
    └── entities/（实体页）
    ↓ Query（提问）
回答（可选：file back 回写）
    ↓ Lint（定期健康检查）
发现矛盾/过期/孤立页 → 修复
    ↓（未来）
合成数据 → 微调 → 知识写入权重

log.md ❌ 未建
